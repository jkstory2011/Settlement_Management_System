**추가로 한 일 (같은 날, 실사용 검증 중 발견)**
- 실사용 확인 중 품목명 뒤에 문구가 더 붙은 경우(예: "...15P" vs "...15P / 품질 보증서 동봉 상품 입니다.")는 완전일치 실패로 매칭이 안 되는 걸 발견(비전스토리 실제 사례로 확인, 원본이어야 할 2,700원 대신 1,950원 적용됨).
- 배치 상세에 **"미확인 후보" 탭** 신설: 화주사를 고르면, 참조표와 완전히 같진 않지만 접두어가 겹치는(품목명이 서로의 접두사인) 후보를 보여주고, "적용" 누르면 그 품목명 그대로 `shipper_item_prices`에 등록 + 이번 배치 라인에 즉시 반영. 자동으로 값을 적용하진 않고 항상 사람이 확인 후 승인하는 방식.
  - 배치 전체(21만 건)를 한 번에 조회하면 8.9초로 timeout 나서, 화주사 단위로 스코프를 좁힘(`batch_unmatched_item_candidates(batch_id, shipper_id)`).
  - 비전스토리로 실제 테스트: 위 사례("휘슬러 모던 커트러리...") 포함 21개 후보 확인, 적용 후 라인의 적용금액이 2,700원으로 정확히 바뀜을 확인.

- 합포장(묶음배송) 판별 방식도 화주사마다 다름을 확인(비전스토리는 품목명에 "$" 대신 "(수량) +" 패턴 사용). `shippers.bundle_pattern`(화주사별 정규식, 없으면 기본값 "$") 컬럼을 추가하고, 화주사 관리 화면에 인라인 입력 칸을 추가. 업로드 시점 계산과 기존 배치 백필(`backfill_is_bundled_chunk`) 모두 화주사별 규칙을 쓰도록 수정, 비전스토리 349건 재판별 확인.
- 라인 조회 탭 "수정" 입력칸이 "최종금액" 위치에 있어서 사용자가 헷갈려함 → "적용금액" 위치로 이동.
- **금액 계산 공식 버그 발견 및 수정** (사장님이 지적: "적용금액+기타운임=최종금액"이어야 하는데 안 맞음):
  - 문제: `applied_amount`(적용금액)에 이미 `기타운임`이 섞여 들어가 있었음(계약단가+기타운임을 한 번에 저장). `final_amount`(최종금액)는 그걸 그대로 썼을 뿐 다시 더하지 않아서, 공식이 "적용금액=계약단가+기타운임, 최종금액=적용금액"으로 어긋나 있었음.
  - 수정: `applied_amount`는 이제 기타운임을 뺀 순수 보정 기본운임(매칭 안 되면 원본 기본운임)만 저장. `final_amount = coalesce(manual_amount, applied_amount) + 기타운임` 공식으로 통일.
  - `final_amount`는 원래 DB 생성 컬럼(GENERATED ALWAYS)이었는데, 공식을 바꾸려면 21만 건 전체를 다시 써야 해서 8초 넘게 걸려 매번 timeout남 → `ALTER COLUMN ... DROP EXPRESSION`으로 일반 컬럼 전환(메타데이터만 바뀌어 0.8초) 후 청크 단위로 백필하는 방식으로 우회.
  - 이 계산에 관여하는 함수 전부 수정: `recompute_batch_applied_amounts_chunk`, `assign_shipper_to_candidate`, `unassign_shipper_candidate`, `update_lines_and_reassign`, `match_return_candidates_to_general`, `set_line_manual_amount`, 업로드 라우트, 미확인후보 적용 라우트. final_amount가 이제 일반 컬럼이라 이 함수들이 앞으로도 매번 직접 계산해서 넣어줘야 함(자동 계산 안 됨).
  - **부수 발견**: `assign_shipper_to_candidate`/`update_lines_and_reassign` 두 함수가 어제(2026-07-03) `shipper_rate_tiers.cj_base_fee`를 타입 기반(`cj_type`)으로 바꿀 때 놓쳐서, 존재하지 않는 컬럼을 참조하는 상태로 방치돼 있었음(호출하면 SQL 에러 발생했을 것) — 이번에 같이 발견해서 `shipper_item_prices` 매칭으로 수정함.
  - 6월 배치 재계산 재실행 → other_fee≠0인 1,715건의 적용금액이 새 공식으로 정정됨, 전체 라인 final_amount 백필 검증 완료.
